Edits files using Hashline anchors from the Read tool.

Usage:
- You must use your `Read` tool at least once in the conversation before editing. This tool will error if you attempt an edit without reading the file. 
- Read output prefixes text lines as `<line>#<hash>|<content>`. Use only the `LINE#ID` portion as `pos` / `end`. Everything after `|` is actual file content.
- The `lines` field must contain real file text, not diff text. Never include Hashline prefixes in `lines`; the tool strips common accidental prefixes, but you should not rely on that.
- Use `edits: [{ op: "replace" | "append" | "prepend", pos?, end?, lines }]`. `replace` requires `pos`; `end` extends the replaced range. `append` / `prepend` may omit anchors to insert at EOF/BOF or create a missing file.
- Multiple edits for the same file should be sent in one call. All anchors are validated against the same pre-edit snapshot and then applied bottom-up.
- ALWAYS prefer editing existing files in the codebase. NEVER write new files unless explicitly required.
- Only use emojis if the user explicitly requests it. Avoid adding emojis to files unless asked.
- The edit will FAIL if any Hashline anchor no longer matches the current file. Re-read the target range and retry with the new `LINE#ID`.
- The edit will FAIL for overlapping ranges, empty anchored inserts, and no-op edits.
- Legacy `oldString` / `newString` / `replaceAll` parameters are not supported.
