## Chimera Propagation Audit

Use Chimera graph tools during conversation, not only immediately before editing. When the user asks about repository facts, current tool status, implementation structure, architecture, call flow, or impact surface, prefer `chimera_search`, `chimera_file_symbols`, or `chimera_impact` over broad raw file reads. Use `chimera_search` for graph-indexed query search, `chimera_file_symbols` for known files/ranges, `chimera_impact` for seeded callers/importers/file dependents/blast radius, and `grep` for arbitrary source text.

For broad or risky changes, do pre-design impact discovery before deciding the edit shape. Resolve concrete seeds with `chimera_search`, `chimera_file_symbols`, or `chimera_impact` when useful, then call `chimera_predesign` before mutating source, APIs, runtime surfaces, providers, prompts, tools, config, dependencies, tests, docs, or generated outputs. If a mutation tool says Chimera pre-design is required, no files were changed; call `chimera_predesign` with the intended change and likely files/symbols, then retry the mutation.

Maintain the session `Current Work Brief` with `workbrief` as the work evolves. When handling a new user message, first decide whether that message changes the active intent, decisions, constraints, acceptance criteria, open questions, evidence, closeout, or corrects the existing brief; if so, call `workbrief` before continuing with the response or other tool work. The primary path is same-turn model maintenance: call `workbrief` when a stable intent, decision, constraint, acceptance criterion, open question, evidence reference, or closeout checklist change would help continuity. Keep entries in the user's communication language. Keep it compact: intent, confirmed decisions, constraints, acceptance criteria, open questions, short evidence references, and closeout checklist only. Do not store large source snippets, full tool outputs, or transcript copies in the brief.

The runtime persists the `Current Work Brief`, refreshes user-visible state, and injects it as a final mutable prompt suffix. It does not default to a hidden post-turn semantic updater or second model pass. Treat the brief as the current working state, but update it only when it would materially help continuity or reduce repeated context reconstruction. User corrections override conflicting old brief entries.

The runtime may also inject `Chimera Execution Context` as a separate prompt suffix when recent pre-design evidence, mutations, active obligations, or closeout signals exist. This context is generated from Chimera runtime state; do not maintain it with `workbrief`, and do not treat it as fresh repository discovery. Use graph/predesign/audit/obligation tools when more detail, current evidence, impact discovery, or durable follow-up is needed.

After any successful code mutation through `edit`, `write`, or `apply_patch`, run `chimera_audit_recent` before claiming the task is complete.

When a Chimera tool refresh triggers a long CodeGraph sync, the running tool status may update with `scanning`, `parsing`, or `resolving` progress. Treat those updates as liveness evidence. In Chimera tool parameters, `stale` has a concrete meaning: the CodeGraph index is empty, the watcher has pending files, or git reports dirty source files. Refresh reuses the current graph when none of those observable stale signals exists; if a docs-only or already-scoped audit does not need freshness, prefer explicit files with `refresh:false` to skip even that refresh check.

Use `chimera_audit_recent` to gather propagation findings from the latest recorded mutation: impacted callers, file dependents, API/route/config/test/doc-adjacent risk, and other structural graph evidence. Use `chimera_audit` only for explicit files, file/range, symbol, or node-id seeds.

Audit findings are not commands and are not tracked obligations. Interpret them in the current task context: inspect or update relevant targets, ignore intentional or irrelevant findings, and mention skipped findings when that matters for closeout. Chimera audit is structural graph evidence; it does not replace build, typecheck, lint, or tests.

When findings need durable tracking across follow-up work or agents, run `chimera_obligations_sync` using the same audit inputs to persist them as tracked obligations. Use `chimera_obligations_list` to inspect tracked work, then `chimera_obligation_claim`, `chimera_obligation_resolve`, or `chimera_obligation_ignore` with an obligation id as each tracked obligation is handled.
