## Chimera Propagation Audit

Use Chimera graph tools during conversation, not only immediately before editing. When the user asks about repository facts, current tool status, implementation structure, architecture, call flow, or impact surface, prefer `chimera_search`, `chimera_file_symbols`, `chimera_context`, or `chimera_impact` over broad raw file reads. Use `chimera_search` for graph-indexed query search, `chimera_file_symbols` for known files/ranges, and `grep` for arbitrary source text.

Maintain the session `Current Work Brief` with `workbrief` as the work evolves. When handling a new user message, first decide whether that message changes the active intent, decisions, constraints, acceptance criteria, open questions, evidence, closeout, or corrects the existing brief; if so, call `workbrief` before continuing with the response or other tool work. The primary path is same-turn model maintenance: call `workbrief` when a stable intent, decision, constraint, acceptance criterion, open question, evidence reference, or closeout checklist change would help continuity. Keep entries in the user's communication language. Keep it compact: intent, confirmed decisions, constraints, acceptance criteria, open questions, short evidence references, and closeout checklist only. Do not store large source snippets, full tool outputs, or transcript copies in the brief.

The runtime persists the `Current Work Brief`, refreshes user-visible state, and injects it as the final mutable prompt suffix. It does not default to a hidden post-turn semantic updater or second model pass. Treat the brief as the current working state, but update it only when it would materially help continuity or reduce repeated context reconstruction. User corrections override conflicting old brief entries.

After any successful code mutation through `edit`, `write`, or `apply_patch`, run `chimera_audit_recent` before claiming the task is complete.

When a Chimera tool refresh triggers a long CodeGraph sync, the running tool status may update with `scanning`, `parsing`, or `resolving` progress. Treat those updates as liveness evidence. In Chimera tool parameters, `stale` has a concrete meaning: the CodeGraph index is empty, the watcher has pending files, or git reports dirty source files. Refresh reuses the current graph when none of those observable stale signals exists; if a docs-only or already-scoped audit does not need freshness, prefer explicit files with `refresh:false` to skip even that refresh check.

Use `chimera_audit_recent` to find propagation obligations from the latest recorded mutation: impacted callers, file dependents, API/route/config/test/doc-adjacent risk, and other places that may need review or updates. Use `chimera_audit` only for explicit files, file/range, symbol, or node-id seeds.

If `chimera_audit_recent` or `chimera_audit` returns candidate obligations, review or update them before finalizing. If an obligation is intentionally skipped, say why. Chimera audit is structural graph evidence; it does not replace build, typecheck, lint, or tests.

When obligations need to be tracked across follow-up work, run `chimera_obligations_sync` using the same audit inputs. Use `chimera_obligations_list` to inspect tracked work, then `chimera_obligation_claim`, `chimera_obligation_resolve`, or `chimera_obligation_ignore` with an obligation id as each obligation is handled.
